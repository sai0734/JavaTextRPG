# 🛡️ THE CHRONICLES OF JAVA : THE EZEN

> **"자바 객체지향의 정수를 담은 텍스트 RPG"**
> 타오르는 고향의 복수를 위해 폭룡 아카르를 찾아 떠나는 용사의 서사를 담은 GUI 기반 RPG 게임입니다.

---

## 1. 프로젝트 설계 의도

* **성격별 대화 및 반응 차이 구현**
    * 동일한 행동(공격, 스킬, 아이템 사용 등)에 대해 캐릭터의 직업이나 성격에 따라 서로 다른 대사와 결과를 출력하도록 설계하였습니다.
    * 이를 통해 사용자에게 보다 생동감 있고 개성 있는 상호작용을 제공하고자 하였습니다.
* **다형성 학습 및 실습**
    * 클래스(`Player`, `Monster`)와 인터페이스(`Skill`)를 기반으로, 다양한 하위 클래스(`Warrior`, `Mage`, `Rogue` 등)가 고유의 행동을 구현하도록 하였습니다.
    * 동일한 메서드 호출이 실제 객체의 타입에 따라 다른 결과를 출력하도록 메서드를 오버로딩 하였으며 "런타임 다형성(Polymorphism)**을 명확히 체험할 수 있도록 구성하였습니다.
* **객체 간 역할 분리 및 책임 분담**
    * **Controller**: `BattleSystem`은 행동 명령을 전달하고 전투 흐름을 제어합니다.
    * **Model**: `Player`와 `Monster` 클래스는 상태 데이터와 핵심 로직을 담당합니다.
    * **Manager**: `Main` 클래스는 전체 게임의 시나리오와 엔트리 포인트를 관리합니다.
    * 이러한 구조를 통해 객체 간 결합도를 낮추고, 각 클래스의 책임을 명확히 구분하였습니다.

## 2. 클래스 다이어그램
*(여기에 직접 제작하신 다이어그램 이미지를 삽입하세요)*

### 🔍 다이어그램 작성 규칙 및 기호 의미
본 프로젝트의 다이어그램은 표준 UML 규격을 준수하여 다음과 같은 의미를 가집니다.

* **필드 접근 제어자 기호**
  * `+` : **public** - 클래스 외부에서 접근 가능
  * `-` : **private** - 해당 클래스 내부에서만 접근 가능
  * `#` : **protected** - 상속받은 자식 클래스에서도 접근 가능
  * `~` : **package(default)** - 동일 패키지 내에서만 접근 가능

* **관계 표현 (선과 화살표 모양)**
  * **상속(Generalization)**: 실선과 속이 빈 화살표(`▷`)를 사용하며 `is-a` 관계를 의미합니다.
  * **실체화(Realization)**: 점선과 속이 빈 화살표를 사용하며 인터페이스의 `implements`를 의미합니다.
  * **연관(Association)**: 실선 화살표(`→`)를 사용하며 클래스가 다른 클래스를 멤버 필드로 가짐(`has-a`)을 의미합니다.
  * **의존(Dependency)**: 점선 화살표를 사용하며 메서드 매개변수 등으로 잠시 사용하는 `uses` 관계를 의미합니다.

## 4. 문제 해결 (Troubleshooting)

### 4.1. 자원 관리 및 아이템 획득 로직의 예외 처리
* **문제**: 스킬 사용 시 MP 부족에 대한 피드백 부재 및 인벤토리 풀(Full) 상태에서 전리품 소실 우려.
* **해결**: 
    * **MP 검증**: 스킬 호출 전 현재 MP를 체크하여 부족 시 로그창(`logArea`)에 즉시 안내 메시지를 출력하고 턴이 넘어가지 않도록 로직을 보호했습니다.
    * **상호작용형 인벤토리 관리**: 전리품 획득 시 인벤토리 용량(10칸)을 체크합니다. 가방이 꽉 찼을 경우, 유저가 기존 리스트에서 삭제할 아이템 번호를 선택하게 하여, 전략적인 아이템 교체가 가능하도록 설계했습니다.

### 4.2. GUI 환경에서의 턴제 상태 동기화
* **문제**: 콘솔 버전의 `while` 루프 제거 후, 매 턴 플레이어와 몬스터의 변동된 HP/MP 상태가 화면에 실시간으로 반영되지 않는 문제.
* **해결**: 
    * **Display Update 분리**: `updateDisplay()` 메서드를 별도로 정의하여 플레이어의 행동(공격, 방어, 아이템 사용 등)이 끝날 때마다 호출하도록 했습니다.
    * **상태 체크 시퀀스**: `플레이어 행동 -> 승리 조건 체크 -> 몬스터 턴 -> 사망 체크 -> 디스플레이 갱신` 순으로 이어지는 이벤트 체인을 구성하여, 턴이 바뀔 때마다 객체의 상태값을 시각적으로 동기화했습니다.

### 4.3. 사용자 입력값 검증 및 예외 핸들링
* **문제**: 기존 `Scanner` 방식에서 숫자가 아닌 문자열 입력 시 `InputMismatchException`이 발생하며 프로그램이 강제 종료되는 현상.
* **해결**: 
    * **입력 방식 전환**: `Scanner` 대신 GUI 환경에 적합한 `JOptionPane.showInputDialog`를 도입했습니다.
    * **예외 캐치 로직**: 유저의 입력값을 `Integer.parseInt()`로 변환하는 과정에서 `try-catch` 블록을 적용했습니다. 숫자가 아닌 값이 들어올 경우 에러 팝업 또는 로그 메시지를 통해 "숫자만 입력 가능함"을 안내하고, 올바른 값이 들어올 때까지 로직을 재수행시켜 프로그램의 안정성을 확보했습니다.

## 5. 사용된 기술 요약
* **Language**: Java 17
* **Library/Framework**: Java Swing (GUI)
* **Design Pattern**: 
    * **Inheritance**: `Player` 및 `Monster` 추상 클래스 설계로 코드 중복 최소화
    * **Interface**: `Skill` 인터페이스를 통한 스킬 구현의 표준화
* **Concurrence**: Swing `Timer`를 활용한 시간차 텍스트 출력 연출

---

## 6. 프로젝트 강점 요약
* **높은 확장성**: 새로운 직업이나 몬스터를 추가할 때 기존 코드를 수정하지 않고 상속만으로 간편하게 확장할 수 있는 유연한 구조를 가집니다.
* **코드 가독성**: 데이터 로직(Logic)과 출력(UI)을 분리하여 유지보수가 용이하며, 명확한 접근 제어자 사용으로 데이터 무결성을 지켰습니다.
* **몰입감 있는 연출**: 단순 텍스트 출력을 넘어 시간차 타이머와 이모지를 활용하여 텍스트 RPG만의 감성을 극대화했습니다.
